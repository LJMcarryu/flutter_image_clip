# 发布流程

## 版本规则

- 破坏性公开 API 或行为变化：提升主版本号。
- 新增兼容能力：提升次版本号。
- 修复缺陷、CI、文档或发布流程：提升 patch 版本号。
- 每次发布必须先更新 `CHANGELOG.md`，并确认 `pubspec.yaml` 与 tag 版本一致。

## 自动发布准备

发布 workflow 使用 pub.dev 的 GitHub Actions OIDC 机制，不保存长期 token。需要在 pub.dev 后台完成一次配置：

1. 打开 `https://pub.dev/packages/flutter_image_clip/admin`。
2. 在 Automated publishing 中启用 GitHub Actions。
3. Repository 填写 `LJMcarryu/flutter_image_clip`。
4. Tag pattern 填写 `v{{version}}`。
5. 在 GitHub repository variables 中设置 `PUB_DEV_AUTOMATED_PUBLISHING=true`。
6. 如果启用环境保护，环境名使用 `pub.dev`，并在 GitHub repository settings 中配置 required reviewers。

相关机制来自 Dart 官方自动发布文档：`https://dart.dev/tools/pub/automated-publishing`。

## 安全检查准备

Dependabot 已覆盖 pub、example pub 和 GitHub Actions 依赖。Dependency Review workflow 在 GitHub Dependency graph 未启用时会以 non-blocking 方式运行；在仓库设置 `Settings -> Code security and analysis` 中启用 Dependency graph 后，可以移除 workflow 中的 `continue-on-error`，把高风险依赖变更改为阻塞项。

## 发布前检查

```sh
flutter pub get
dart format --set-exit-if-changed lib test benchmark tool example/lib example/integration_test
flutter analyze
flutter test
dart run tool/check_api_snapshot.dart
dart run tool/check_public_api_docs.dart
dart run benchmark/image_processor_benchmark.dart --check benchmark/baseline.json
dart doc --output doc/api
dart pub publish --dry-run
```

## 推荐发布顺序

```sh
git status --short
git tag v0.7.1
git push origin main
git push origin v0.7.1
```

tag 推送后，`Release checks` 会先跑格式化、分析、测试、API snapshot、公开 API 文档检查、benchmark、dartdoc 和 pub dry-run。只有全部通过且 `PUB_DEV_AUTOMATED_PUBLISHING=true` 时，`Publish to pub.dev` job 才会执行真实发布。

如果 pub.dev OIDC 尚未启用，自动发布 job 会失败。此时不要重新推同一个 tag；先启用 pub.dev 自动发布，或者在确认 release checks 全绿后本地执行：

```sh
dart pub publish --force
```

## 发布失败处理

- 已上传到 pub.dev 的版本不能覆盖。
- 如果发布后发现 CI 或 release checks 失败，立即修复并发布下一个 patch 版本。
- 不建议依赖 unpublish；pub.dev 只在极少数情况下允许删除已发布包。
- GitHub Release notes 应与 `CHANGELOG.md` 对齐。
