# 仓库信任与安全配置

这些配置有一部分必须在 GitHub 或 pub.dev 后台完成，无法通过代码提交直接打开。仓库内的 CI 和文档已经按成熟三方库的默认策略准备好。

## pub.dev verified publisher

目标：让 pub.dev 不再显示 `unverified uploader`，提高使用者对发布来源的信任。

操作步骤：

1. 准备一个可验证域名，例如项目或组织域名。
2. 在 pub.dev 登录发布账号，进入 publisher 管理页面。
3. 创建 verified publisher，并按 pub.dev 要求完成域名 DNS 验证。
4. 把 `flutter_image_clip` 转移到该 publisher。
5. 发布前确认包页面显示 verified publisher。

## GitHub Dependency graph

目标：让 Dependency Review 在 PR 中成为阻塞项，防止高风险依赖变更进入主分支。

操作步骤：

1. 打开 GitHub 仓库 `Settings -> Code security and analysis`。
2. 启用 `Dependency graph`。
3. 启用 `Dependabot alerts`。
4. 保持 `.github/workflows/security.yml` 中的 Dependency Review job 为 blocking。
5. 在分支保护规则中把 `Dependency review` 加入 required status checks。

## pub.dev 自动发布

目标：发布不依赖本地长期 token，改用 GitHub Actions OIDC。

操作步骤：

1. 打开 `https://pub.dev/packages/flutter_image_clip/admin`。
2. 在 Automated publishing 中启用 GitHub Actions。
3. Repository 填写 `LJMcarryu/flutter_image_clip`。
4. Tag pattern 填写 `v{{version}}`。
5. 在 GitHub repository variables 中设置 `PUB_DEV_AUTOMATED_PUBLISHING=true`。
6. 如启用 GitHub environment protection，环境名保持 `pub.dev`。

## 分支保护建议

建议把以下检查设置为 `main` 的 required status checks：

- `Flutter checks (3.38.1)`
- `Flutter checks (stable)`
- `Android integration test (3.38.1)`
- `Android integration test (stable)`
- `iOS example build`
- `iOS integration test`
- `Dependency review`

## 发布前确认

- pub.dev 包页显示 verified publisher。
- Dependency Review 对测试 PR 是 blocking。
- release tag 的 `Release checks` 全绿。
- 如果 `PUB_DEV_AUTOMATED_PUBLISHING=true`，确认 `Publish to pub.dev` job 进入受保护环境并由授权人审批。
